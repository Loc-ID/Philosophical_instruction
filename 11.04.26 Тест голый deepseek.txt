Вот три основных сценария, почему это происходит, и что с этим делать:

🧠 Причина 1: Браузер «замораживает» неактивные вкладки
Это самая вероятная причина. Современные браузеры агрессивно оптимизируют использование памяти. Когда вы уходите на другую вкладку, браузер может приостановить работу JavaScript на «фоновой» вкладке или даже полностью выгрузить её состояние из памяти. Когда вы возвращаетесь, страница перезагружается, и всё React-состояние (useState, useReducer) сбрасывается к исходным значениям.

Это поведение можно отследить через вкладку "Performance" в DevTools (инструменты разработчика), где видна остановка выполнения задач на время неактивности вкладки.

🔄 Причина 2: Условный рендеринг и размонтирование компонента (Ошибка в логике приложения)
Вторая по распространённости причина — это то, как реализован интерфейс с вкладками внутри самого приложения. Очень часто для переключения контента используют условный рендеринг:

javascript
{activeTab === 'form' && <MyForm />}
Проблема в том, что при переключении на другую вкладку (activeTab меняется), компонент <MyForm /> не просто скрывается — он полностью размонтируется (удаляется из DOM). Всё его внутреннее состояние безвозвратно теряется. Когда пользователь возвращается, компонент создаётся заново с чистым состоянием. Это нормальная и ожидаемая работа React.

🚀 Причина 3: Новое поведение в React 19 (компонент <Activity>)
Если вы используете React 19.2 или новее, в приложении может использоваться новый компонент <Activity>. Он специально создан для решения проблемы с сохранением состояния в неактивных частях UI.

Если компонент обёрнут в <Activity mode="hidden">, его состояние и DOM сохраняются, но все эффекты (useEffect, подписки) приостанавливаются. При возврате в режим mode="visible" всё возобновляется с того же места. Проблема может возникнуть, если вы ожидаете, что состояние сбросится (как при обычном размонтировании), а оно сохраняется, или наоборот — используете <Activity> не по назначению.

🛠️ Как это исправить: проверенные решения
Сохранение состояния в URL: Самый надёжный и «веб-дружественный» способ. Сохраняйте состояние формы (например, номер страницы, фильтры) в параметрах запроса (?step=2). При возвращении на вкладку состояние будет восстановлено из URL.

Использование sessionStorage: Если данные формы не должны влиять на URL, сохраняйте их в sessionStorage. Данные в нём живут, пока открыта вкладка, и переживут её «заморозку» браузером.

CSS-скрытие вместо условного рендеринга: Вместо того чтобы размонтировать компонент вкладки, скрывайте его с помощью CSS (display: none). Это сохранит DOM и состояние, но может увеличить потребление памяти, так как все скрытые компоненты останутся в дереве.

Подъём состояния (Lifting State Up): Если состояние должно сохраняться при переключении между связанными компонентами, храните его в ближайшем общем родителе, который не размонтируется. Это классический паттерн React.

Метод: POST или GET, но URL лучше передавать в теле JSON при POST-запросе.

Ограничения: Таймаут соединения, максимальный размер загружаемого контента (чтобы не исчерпать память сервера), проверка схемы URL (только http и https).

Потоковая загрузка: Чтобы не держать весь файл в оперативной памяти, мы будем читать ответ чанками и суммировать их длину.

python
from flask import Flask, request, jsonify
import requests
from urllib.parse import urlparse

app = Flask(__name__)

# Настройки безопасности
MAX_CONTENT_LENGTH = 50 * 1024 * 1024  # 50 MB максимум
REQUEST_TIMEOUT = 10  # секунд на соединение и чтение
ALLOWED_SCHEMES = {'http', 'https'}

@app.route('/get-size', methods=['POST'])
def get_content_size():
    # 1. Получаем URL из тела запроса
    data = request.get_json(silent=True)
    if not data or 'url' not in data:
        return jsonify({'error': 'URL is required'}), 400

    url = data['url'].strip()
    if not url:
        return jsonify({'error': 'URL cannot be empty'}), 400

    # 2. Проверяем схему URL (защита от file://, ftp:// и пр.)
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        return jsonify({'error': f'Only {", ".join(ALLOWED_SCHEMES)} URLs are allowed'}), 400

    # 3. Выполняем запрос с потоковой загрузкой
    try:
        # stream=True не загружает всё тело сразу
        response = requests.get(
            url,
            timeout=REQUEST_TIMEOUT,
            stream=True,
            headers={'User-Agent': 'Flask-Size-Checker/1.0'}  # Хороший тон представляться
        )
        response.raise_for_status()  # выбросит исключение при 4xx/5xx

        total_bytes = 0
        # 4. Скачиваем чанками и контролируем размер
        for chunk in response.iter_content(chunk_size=8192):
            total_bytes += len(chunk)
            if total_bytes > MAX_CONTENT_LENGTH:
                # Прерываем загрузку, если превышен лимит
                response.close()
                return jsonify({'error': f'Content exceeds maximum allowed size of {MAX_CONTENT_LENGTH} bytes'}), 413

        # Закрываем соединение явно (хорошая практика)
        response.close()

        return jsonify({
            'url': url,
            'size_bytes': total_bytes,
            'content_type': response.headers.get('Content-Type', 'unknown')
        })

    except requests.exceptions.Timeout:
        return jsonify({'error': 'Request timed out'}), 504
    except requests.exceptions.ConnectionError:
        return jsonify({'error': 'Failed to connect to the URL'}), 502
    except requests.exceptions.HTTPError as e:
        return jsonify({'error': f'HTTP error: {e.response.status_code}'}), 502
    except requests.exceptions.RequestException as e:
        return jsonify({'error': f'Request failed: {str(e)}'}), 500

if __name__ == '__main__':
    app.run(debug=True)

Вот ключевая причина и способы решения.

❓ Почему append удалили?
Причина кроется в производительности. Метод append работал не как встроенный в Python list.append(), который изменяет существующий объект, а создавал полностью новый DataFrame при каждом добавлении.

В результате код вроде этого:

python
for row in data:
    df = df.append(row) 
...приводил к огромным накладным расходам, так как на каждой итерации в памяти создавалась новая, всё более крупная копия DataFrame. Это очень неэффективно, поэтому разработчики библиотеки приняли решение удалить этот метод.

✅ Как исправить: 3 проверенных способа
Вместо df.append() теперь официально рекомендуется использовать pd.concat().

1. Использование pd.concat() (универсальный способ)
Это самый надёжный метод для добавления одной или нескольких строк из словаря, списка или другого DataFrame.

python
import pandas as pd

# Ваш исходный DataFrame
df = pd.DataFrame(columns=['A', 'B'], data=[[1, 2]])

# Новая строка, которую нужно добавить
new_row = {'A': 3, 'B': 4}

# Создаём из словаря DataFrame и объединяем
df = pd.concat([df, pd.DataFrame([new_row])], ignore_index=True)
print(df)
2. Использование loc для одной строки (только для RangeIndex)
Если вы добавляете одну строку и используете стандартный числовой индекс, можно применить более краткую запись.

python
df.loc[len(df)] = new_row
Этот способ проще, но работает только в том случае, если у вашего DataFrame индекс по умолчанию (RangeIndex). Если индекс был изменён, этот метод может привести к ошибке или перезаписи существующей строки.

3. Сбор данных в список и создание DataFrame в конце (самый эффективный метод)
Этот метод не является прямой заменой append, но его важно знать. Если вы добавляете строки в цикле, создавать новый DataFrame на каждой итерации — худшее, что можно сделать. Гораздо правильнее сначала накопить все данные в списке Python, а потом один раз создать из него DataFrame.

python
data = []
for item in some_large_iterable:
    # Собираем строки в виде словарей или списков
    data.append({'A': item.value1, 'B': item.value2})

# Создаём итоговый DataFrame за один шаг
df = pd.DataFrame(data)

Для финтех-стартапа, работающего с финансовыми транзакциями, оптимальным и самым безопасным выбором будет PostgreSQL. Выбор базы данных в этой сфере — это не вопрос технологических трендов, а вопрос гарантий безопасности и юридических последствий.

Вот ключевые причины, по которым PostgreSQL подходит для этой задачи лучше, чем MongoDB.

🛡️ ACID и целостность данных: Фундаментальная разница
Главное требование к любой финансовой системе — строжайшая согласованность данных. Деньги не могут исчезнуть или появиться из ниоткуда.

PostgreSQL — это реляционная база данных, которая десятилетиями доказывала свою надежность. Она полностью поддерживает ACID (атомарность, согласованность, изолированность, долговечность), обеспечивая бескомпромиссную целостность данных при любых, даже самых сложных, операциях. Это стандарт индустрии для банковских и финансовых систем.

MongoDB — документоориентированная NoSQL-база. Несмотря на то, что с версии 4.0 она также поддерживает ACID-транзакции, её архитектура изначально не была спроектирована для этого. Транзакции в MongoDB — это дополнительная возможность, которая может быть менее производительной и иметь свои нюансы в распределенных кластерах. Такой подход несет в себе риски, неприемлемые для финансов.

Исследования показывают, что миграция с MongoDB на реляционные БД в финтехе может снизить среднее время отклика на 42% и повысить согласованность данных до 100%.

🏗️ Сложные связи и запросы: Язык финансов
Финансовые данные по своей природе реляционные: у пользователя есть несколько счетов, к ним привязаны транзакции, к транзакциям — статусы и т.д.

PostgreSQL с помощью SQL позволяет легко и эффективно строить такие связи через JOIN, агрегировать данные, создавать сложные финансовые отчеты и проводить аудит.

MongoDB плохо подходит для операций JOIN. Её сильная сторона — работа с денормализованными документами. Попытка реализовать сложные связи в MongoDB часто приводит к избыточности данных и громоздким операциям $lookup, которые уступают SQL-запросам.

⚖️ Строгая схема данных: Основа безопасности и надежности
В финансовой сфере строгая типизация данных — не прихоть, а необходимость.

PostgreSQL требует заранее определенной и строгой схемы данных (schema). Каждое поле в таблице имеет свой тип (DECIMAL, TIMESTAMP, VARCHAR и т.д.). Это исключает возможность записи некорректных данных и служит первой линией защиты от ошибок и потенциальных уязвимостей.

MongoDB известна своей гибкой, бессхемной архитектурой. Это огромный плюс для быстрой разработки, но критический недостаток для систем, где важна строгость и предсказуемость. Нельзя допустить, чтобы сумма транзакции была записана как строка или дата — как дата в неверном формате.

📊 Масштабирование и производительность
PostgreSQL традиционно масштабируется вертикально (увеличением мощности сервера), но может и горизонтально с помощью таких расширений, как Citus. Финтех-проекты часто начинаются с вертикального масштабирования, и мощности одного мощного сервера с PostgreSQL хватает для обработки десятков тысяч транзакций в секунду. Производительность PostgreSQL предсказуема и отлично оптимизирована для сложных запросов.

MongoDB изначально создавалась для горизонтального масштабирования (шардирования), что делает её отличным выбором для работы с огромными объемами неструктурированных данных. Однако в мире транзакций на первый план выходит не объем данных, а их целостность.

💎 Что в итоге: Практический вердикт
Для финтех-стартапа с фокусом на пользовательские транзакции PostgreSQL — это осознанный и единственно верный выбор.

Это не значит, что MongoDB — плохая база данных. Просто у неё другая область применения. В архитектуре вашего стартапа она может занять своё место, но не в роли основного хранилища транзакций.

MongoDB: вспомогательное хранилище для менее критичных данных, где важна гибкость и скорость разработки: профили пользователей, история действий, логи, кэш аналитики.

🏗️ High-Level Design (Высокоуровневая архитектура)
Поток данных в системе будет выглядеть следующим образом:

API Gateway: Принимает запросы на отправку уведомлений от ваших внутренних сервисов (например, при создании заказа или изменении статуса).

Orchestrator Service: «Мозг» системы, который обогащает запрос данными о пользователе и его предпочтениями.

Message Broker (Кроличья нора): Центральная нервная система, которая разделяет поток на очереди и гарантирует, что ни одно уведомление не потеряется при пиковых нагрузках или сбоях.

Priority Queues (Очереди с приоритетом): Разделяют уведомления по важности.

Channel Workers: Отдельные легковесные сервисы для каждого канала (Email, SMS, Push, In-App), которые можно масштабировать независимо друг от друга.

3rd-Party Providers (Внешние провайдеры): Сервисы для отправки писем, SMS и пушей.

🧩 Ключевые компоненты и бюджетные решения
1. Приоритизация и очереди (Сердце системы)
Ваше требование разделить транзакционные и маркетинговые уведомления — это классическая задача для очередей сообщений.

Почему RabbitMQ?: В отличие от Apache Kafka, который оптимизирован для хранения огромных потоков данных, RabbitMQ изначально поддерживает приоритетные очереди (x-max-priority), что идеально подходит для вашей задачи. Он позволит рабочим сервисам всегда сначала разбирать сообщения из очереди с высоким приоритетом, а маркетинговые рассылки обрабатывать в фоне.

💸 Бюджетное решение: Вы можете развернуть RabbitMQ самостоятельно на вашем сервере или в Kubernetes-кластере, что будет значительно дешевле, чем платные облачные очереди (например, Amazon MQ).

2. Гарантированная доставка и Retry (Система защиты)
Гарантия at-least-once: RabbitMQ с подтверждениями (acknowledgements) гарантирует, что сообщение не будет удалено из очереди, пока воркер не подтвердит его успешную обработку.

Политика Retry с Backoff: Если провайдер (например, Twilio) возвращает ошибку, воркер не подтверждает сообщение. Используйте экспоненциальную задержку с добавлением случайного джиттера (exponential backoff with jitter) перед повторной попыткой, чтобы не «завалить» провайдера повторными запросами и дать ему время на восстановление.

💸 Бюджетное решение: Эту логику легко реализовать в коде ваших воркеров.

3. Устойчивость при падении провайдера
Dead Letter Queue (DLQ) и Fallback: Установите лимит повторных попыток (например, 5). Если уведомление не удалось доставить, оно перемещается в DLQ. Дальше можно:

Circuit Breaker: Воркер автоматически прекращает попытки отправки через «упавший» канал на некоторое время, чтобы не тратить ресурсы впустую.
Fallback-стратегия: Настройте правило в оркестраторе: если, например, push-уведомление не доставлено, отправить пользователю In-App сообщение или SMS.
💸 Бюджетное решение: Хранение неудачных сообщений в отдельной очереди RabbitMQ практически не требует затрат и позволяет вручную или автоматически переложить их в основную очередь после восстановления провайдера.

4. Хранение данных и пользовательские настройки
💾 База данных: PostgreSQL — отличный выбор для хранения истории уведомлений, статусов доставки и, что самое важное, пользовательских предпочтений (каналы связи, типы уведомлений, «тихие часы»). Это стандартное и недорогое решение.

⚡ Кэш: Redis необходим для молниеносной проверки статуса и настроек пользователя в оркестраторе, чтобы не нагружать основную базу данных при каждом событии.

💸 Бюджетное решение: И PostgreSQL, и Redis можно развернуть самостоятельно или использовать их недорогие managed-версии у облачных провайдеров.

5. Обработка каналов и воркеры
Каждый канал — это отдельный легковесный сервис, который можно написать на чём угодно (Node.js, Python, Go) и масштабировать только тогда, когда это нужно.

📧 Email Worker: Использует AWS SES или SendGrid для отправки. У этих сервисов очень низкая стоимость за тысячу писем (SES: ~$0.10/1000), что идеально для стартапа.

ии:Выше мои мысли, далее мой ответ:

Pre-commit хуки — это скрипты, которые Git запускает перед созданием коммита. Они работают как автоматический фильтр: если проверка не пройдена, коммит блокируется, и проблемный код не попадает в репозиторий.

Способы настройки pre-commit хуков
1. «Ручной» способ: напрямую через скрипты
Самый простой способ — разместить исполняемый скрипт в каталоге .git/hooks/ вашего репозитория:

Создайте файл .git/hooks/pre-commit (без расширения)

Сделайте его исполняемым: chmod +x .git/hooks/pre-commit

Пример скрипта для JavaScript-проекта:

bash
#!/bin/sh
echo "Running pre-commit checks..."

# Запускаем линтинг
npm run lint
if [ $? -ne 0 ]; then
    echo "Linting failed. Please fix errors before committing."
    exit 1
fi

# Запускаем тесты
npm run test
if [ $? -ne 0 ]; then
    echo "Tests failed. Please fix failing tests before committing."
    exit 1
fi

echo "All checks passed!"
exit 0
Важные моменты:

Если скрипт возвращает код 0 — коммит разрешен.

Если возвращает 1 — коммит блокируется.

2. Фреймворк pre-commit: стандарт индустрии
pre-commit — это Python-пакет, который упрощает управление хуками, особенно в командной работе. Он хранит всю конфигурацию в файле .pre-commit-config.yaml прямо в репозитории, что позволяет легко делиться настройками с коллегами.

Установка:

bash
pip install pre-commit
Создание конфигурации:
Создайте файл .pre-commit-config.yaml в корне проекта. Пример для Python-проекта:

yaml
repos:
  # Базовые проверки для любых проектов
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.6.0
    hooks:
      - id: trailing-whitespace   # Удаляет пробелы в конце строк
      - id: end-of-file-fixer     # Добавляет пустую строку в конец файла
      - id: check-yaml            # Проверяет синтаксис YAML-файлов
      - id: check-added-large-files  # Запрещает коммит больших файлов

  # Форматирование и линтинг Python
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.9
    hooks:
      - id: ruff
        args: ["--fix"]           # Автоматически исправляет ошибки
      - id: ruff-format           # Форматирует код
Активация:

bash
pre-commit install
После этого хуки будут автоматически запускаться при каждом git commit.

Популярные pre-commit хуки
Линтинг и форматирование: ESLint (JS), Flake8/Pylint (Python), Black (Python)

Проверка секретов: ggshield (интегрируется с GitGuardian API), gitleaks

Защита веток: no-commit-to-branch — блокирует коммиты в main/master

Проверка сообщений коммитов: commitlint для соблюдения Conventional Commits

Как поделиться хуками с командой
В отличие от «ручного» способа, где файлы лежат в .git/hooks (эта папка не попадает в репозиторий), фреймворк pre-commit хранит настройки в файле .pre-commit-config.yaml, который можно закоммитить.

Для JavaScript/TypeScript проектов есть альтернатива — Husky, который устанавливает хуки через package.json, что упрощает настройку в Node.js окружении.

Советы по эффективному использованию
Держите хуки быстрыми: Медленные хуки раздражают. Тяжелые проверки (например, полный прогон тестов) лучше оставить для CI/CD.

Используйте pre-commit run --all-files: При первом внедрении полезно прогнать все файлы через новые правила.

Обход хуков (с осторожностью!): git commit --no-verify — пропускает все хуки.

Пропуск отдельных хуков: SKIP=ruff git commit -m "message" — пропускает только указанный хук.

Если появятся вопросы по конкретным хукам для вашего стека технологий, обращайтесь.

🔐 Современный золотой стандарт: HttpOnly Cookie
Большинство экспертов сходятся во мнении, что хранение JWT в HttpOnly Cookie — это лучший баланс между безопасностью и удобством для большинства веб-приложений.

Защита от XSS: Флаг HttpOnly делает cookie недоступными для JavaScript, что полностью нейтрализует самый распространенный вектор атаки для кражи токенов.

Автоматическая отправка: Браузер сам добавляет cookie к каждому запросу, что упрощает код фронтенда.

Дополнительные флаги безопасности:

Secure: Гарантирует, что cookie будут передаваться только по HTTPS, защищая от перехвата данных.

SameSite=Strict или Lax: Ключевой механизм защиты от CSRF-атак. Strict полностью блокирует отправку cookie с любого стороннего сайта, Lax разрешает отправку только для безопасных HTTP-методов (например, GET) при переходе с другого сайта.

⚠️ Важно: При использовании HttpOnly Cookie на бэкенде все равно необходимо настроить CORS правильно (с Access-Control-Allow-Credentials: true), если фронтенд и бэкенд находятся на разных доменах.

🧠 In-Memory: выбор для максимальной безопасности
Для приложений, где цена ошибки очень высока (например, финансовые сервисы), лучшим выбором будет хранение токена исключительно в памяти (in-memory).

Как это работает: После успешной аутентификации токен сохраняется в обычную JavaScript-переменную (например, в состояние React-компонента или глобальное хранилище вроде Pinia). Это означает, что токен существует только до тех пор, пока открыта текущая вкладка браузера.

Главный недостаток: Потеря сессии при обновлении страницы или открытии новой вкладки, что крайне неудобно для пользователя.

Элегантное решение: Чтобы сохранить и безопасность, и удобство, часто используется гибридный подход:

Access Token (короткоживущий токен доступа) хранится в памяти.
Refresh Token (долгоживущий токен для обновления) хранится в HttpOnly Cookie.
Когда Access Token истекает, приложение автоматически, не спрашивая пользователя, запрашивает новый, используя Refresh Token из cookie. Это называется "тихая аутентификация" (silent authentication).
🧬 Продвинутая архитектура: BFF (Backend for Frontend)
Это паттерн, при котором между фронтендом и микросервисами появляется специальный сервер — BFF. Его задача — взять на себя всю работу с аутентификацией и хранением токенов.

Фронтенд отправляет учетные данные пользователя на BFF.

BFF обменивает их на JWT и хранит его у себя на сервере.

Фронтенду BFF выдает простую сессионную cookie, которая и используется для всех последующих взаимодействий.

Преимущества: Токен JWT вообще не попадает в браузер, что полностью исключает риск его кражи через клиентские уязвимости. Это один из самых безопасных подходов на сегодняшний день.

Оба формата — хорошие инструменты для работы с конфигурациями, но они решают немного разные задачи. У каждого есть свои сильные стороны и «подводные камни». Ключевое различие — в балансе между гибкостью и строгостью. Давайте разберем их по основным критериям, чтобы вам было проще сделать выбор.

⚖️ Сравнение YAML и TOML: Ключевые различия
1. Читаемость и удобство для человека
Оба формата созданы быть более удобными для человека, чем JSON, но идут к этому разными путями.

YAML: Стремится к минимализму в синтаксисе, используя отступы (как в Python) для обозначения вложенности, а не скобки. Это делает его очень чистым и похожим на естественный язык, что снижает порог входа. Он идеален для сложных, глубоко вложенных структур, так как сохраняет высокую читаемость без визуального шума из скобок.

TOML: Синтаксис основан на INI-файлах: он использует [секции] и пары ключ = "значение". Это делает структуру очень явной и понятной. Вложенность обозначается точками в имени секции (напр., [database.credentials]), что для многих разработчиков интуитивно понятнее, чем подсчет пробелов.

2. Надежность и предсказуемость
Это, пожалуй, самое важное различие, которое напрямую влияет на стабильность работы приложения.

YAML: Его главный недостаток — чрезмерная гибкость, которая часто приводит к скрытым ошибкам. К ним относятся:

Чувствительность к отступам: Ошибка в один пробел может сломать всю структуру, и эту ошибку сложно заметить визуально.

Проблема «Норвегии»: YAML может интерпретировать строки как другие типы данных. Например, значение country: NO будет распознано как булево false (поскольку NO — это стандартное значение для false в YAML 1.1), а version: 1.10 может стать числом с плавающей точкой 1.1.

Риски безопасности: Небезопасная загрузка YAML (например, через yaml.load в Python) может привести к выполнению произвольного кода. Всегда нужно использовать yaml.safe_load.

TOML: Создавался как прямая альтернатива этим проблемам YAML. Он строг, однозначен и предсказуем.

Нет неявных преобразований: Все строки должны быть в кавычках, поэтому country = "NO" всегда будет строкой, а married = false — явным булевым значением.

Нет зависимости от отступов: Структура определяется секциями, что делает формат очень устойчивым к ошибкам форматирования.

3. Производительность
Здесь разрыв очевиден.

YAML: Из-за сложности и гибкости своего синтаксиса YAML является одним из самых медленных форматов для парсинга. Бенчмарки показывают, что он может быть в 10-100 раз медленнее, чем JSON или бинарные форматы.

TOML: Будучи намного проще, TOML парсится значительно быстрее, чем YAML, и часто сравним по скорости с JSON.

4. Экосистема и поддержка
YAML: Это безусловный стандарт в мире DevOps и облачных технологий. Его используют Kubernetes, Docker Compose, Ansible, GitHub Actions и многие другие инструменты. У него огромная экосистема, множество парсеров и утилит (например, yq для работы с YAML в командной строке).

TOML: Набирает популярность как стандарт для конфигурации самих приложений и пакетных менеджеров. Он стал официальным форматом для pyproject.toml в Python (PEP 518/621), используется в Rust (Cargo), Go (многие проекты) и Poetry. Его поддержка растет, и он уже встроен в стандартную библиотеку Python 3.11+ (tomllib).

🎯 Сравнение с образцом (Literal Patterns)
Самый простой случай — сопоставление с константами (числа, строки, True, False, None):

python
def get_http_status_message(code):
    match code:
        case 200:
            return "OK"
        case 404:
            return "Not Found"
        case 500:
            return "Internal Server Error"
        case _:
            return f"Unknown status code: {code}"

print(get_http_status_message(404)) # Вывод: Not Found
print(get_http_status_message(418)) # Вывод: Unknown status code: 418
Здесь case _ — это "джокер" (wildcard), который срабатывает, если ни один из предыдущих образцов не подошёл, и выполняет роль блока default.

🧩 Сопоставление с последовательностями (Sequence Patterns)
Одна из самых мощных возможностей — деструктуризация данных. Вы можете проверить длину и структуру последовательности и "распаковать" её элементы в переменные.

python
def process_command(command):
    match command.split():
        case ["quit"]:
            print("Goodbye!")
            return True
        case ["go", direction]:
            print(f"Moving {direction}.")
        case ["pick", "up", obj]:
            print(f"You picked up the {obj}.")
        case ["attack", target, "with", weapon]:
            print(f"Attacking {target} using {weapon}.")
        case _:
            print(f"Sorry, I don't understand '{' '.join(command)}'.")

process_command("go north")     # Moving north.
process_command("pick up sword") # You picked up the sword.
process_command("quit")         # Goodbye!
case ["go", direction] сопоставляется только с последовательностью из ровно двух элементов, и если это так, связывает второй элемент с переменной direction.

🔗 Комбинирование образцов (OR Patterns) и условия (Guards)
Для большей гибкости можно использовать | для объединения нескольких образцов и if для добавления дополнительных условий.

python
def categorize_number(num):
    match num:
        case 0:
            return "Zero"
        case 1 | 2 | 3:
            return "Small positive"
        case int(n) if n > 0:
            return f"Positive integer: {n}"
        case int(n) if n < 0:
            return f"Negative integer: {n}"
        case _:
            return "Not an integer"

print(categorize_number(2))   # Small positive
print(categorize_number(42))  # Positive integer: 42
print(categorize_number(-10)) # Negative integer: -10
Здесь case int(n) if n > 0 сопоставляется с любым целым числом, связывает его с переменной n, а затем проверяет условие n > 0.

🗺️ Сопоставление со словарями (Mapping Patterns)
Вы можете проверять наличие и значения ключей в словарях.

python
def process_event(event):
    match event:
        case {"type": "click", "x": x, "y": y}:
            print(f"Click at ({x}, {y})")
        case {"type": "keypress", "key": key, "modifiers": mods}:
            print(f"Key '{key}' pressed with modifiers: {mods}")
        case {"type": "unknown"}:
            print("Received unknown event type")
        case _:
            print("Invalid event structure")

process_event({"type": "click", "x": 100, "y": 200})
# Вывод: Click at (100, 200)
process_event({"type": "keypress", "key": "A", "modifiers": ["Shift"]})
# Вывод: Key 'A' pressed with modifiers: ['Shift']
Обратите внимание: case {"type": "click", "x": x, "y": y} сработает даже если в словаре будут дополнительные ключи, которых нет в образце.

💎 Ключевые преимущества match/case
Выразительность: Код становится чище и легче для чтения, особенно при работе со сложными структурами данных.

Безопасность: В отличие от switch в C, здесь нет "проваливания" (fall-through) — выполняется только один блок case.

Мощь: match/case позволяет одновременно проверять тип, структуру и значения данных, извлекая нужные компоненты в переменные.

🐍 Пример функции с :=
Допустим, у нас есть файл с логами, и мы хотим собрать все строки, содержащие слово "ERROR", но при этом экономно читать файл построчно.

python
def get_error_lines(filepath):
    """Возвращает список строк с 'ERROR' из файла, используя walrus operator."""
    errors = []
    with open(filepath, 'r') as f:
        # Классический цикл чтения строк файла с :=
        while line := f.readline().strip():
            if "ERROR" in line:
                errors.append(line)
    return errors

💡 Где ещё полезен :=
Оператор := позволяет присвоить значение переменной внутри выражения, что особенно удобно в условиях if или while.

Пример в условии if:

python
# Без walrus: вычисляем дважды
data = get_data_from_slow_api()
if data:
    process(data)

# С walrus: вычисляем один раз и сразу проверяем

🚦 Коды статусов HTTP — ваши союзники
Важно: JSON-тело ошибки не заменяет HTTP-статус. Они работают в паре. Например:

200 OK + { "success": true, ... }

400 Bad Request + { "success": false, "error": {...} }

404 Not Found + { "success": false, "error": {...} }

Нельзя отправлять ошибку с кодом 200 — это нарушает семантику HTTP и усложняет мониторинг.

На примере FastAPI / Starlette:
Вы можете зарегистрировать глобальный обработчик исключений, который будет перехватывать любые необработанные исключения и преобразовывать их в JSON.

from fastapi.responses import JSONResponse

@app.exception_handler(Exception)

    # Логируем реальную ошибку (но не возвращаем пользователю)
    logger.error(f"Unhandled exception: {exc}", exc_info=True)
    
    # Возвращаем безопасный JSON для клиента
    return JSONResponse(
        status_code=500,
        content={
            "success": False,
            "error": {
                "code": "INTERNAL_SERVER_ERROR",
                "message": "An unexpected error occurred. Please try again later."
            }
        }
    )

Аналогичный подход через @app.errorhandler(Exception).

🚫 Обработка ситуаций вне контроллера
Жёсткое требование JSON означает, что вы должны контролировать ответ даже тогда, когда запрос не доходит до вашего кода:

Некорректный JSON в теле запроса:

📝 Пограничные случаи
Редиректы (3xx): Обычно их обрабатывает браузер или HTTP-клиент. Возвращать JSON с кодом 302 бессмысленно, так как тело ответа часто игнорируется. В REST API редиректы используются редко; лучше вернуть 200 с URL в поле data.

Пустой ответ (204 No Content): Допустим при успешном DELETE. Тело отсутствует, и это нормально. Чтобы соблюсти «только JSON», можно возвращать 200 с { "success": true }, но 204 более RESTful.

python
from fastapi import FastAPI, Query, HTTPException
from fastapi.responses import StreamingResponse
import csv
import io
from typing import List, Optional
from pydantic import BaseModel

app = FastAPI()

# Модель данных пользователя (пример)
class UserOut(BaseModel):
    id: int
    email: str
    full_name: str
    created_at: str

# Заглушка получения данных из БД
async def fetch_users(limit: int = 1000) -> List[UserOut]:
    # Здесь реальный запрос к БД
    return [
        UserOut(id=1, email="alice@example.com", full_name="Alice Smith", created_at="2023-01-15"),
        UserOut(id=2, email="bob@example.com", full_name="Bob Johnson", created_at="2023-02-20"),
    ]

@app.get("/export/users.csv")
async def export_users_csv(
    limit: int = Query(1000, le=10000, description="Max records to export"),
):
    """
    Экспорт данных пользователей в CSV.
    Возвращает файл `users_export.csv`, готовый к открытию в Excel.
    """
    try:
        users = await fetch_users(limit=limit)
    except Exception as e:
        raise HTTPException(status_code=500, detail="Database query failed")

    # Создаём CSV в памяти
    output = io.StringIO()
    writer = csv.writer(output, delimiter=';')  # для Excel под Windows часто используют ';'

    # Заголовки столбцов
    writer.writerow(["ID", "Email", "Full Name", "Created At"])

    # Данные
    for user in users:
        writer.writerow([user.id, user.email, user.full_name, user.created_at])

    # Перемещаем указатель в начало
    output.seek(0)

    # Возвращаем потоковый ответ
    return StreamingResponse(
        output,
        media_type="text/csv",
        headers={
            "Content-Disposition": "attachment; filename=users_export.csv"
        }
    )
🧩 Важные детали для совместимости с Excel
Разделитель: По умолчанию Excel ожидает разделитель, соответствующий региональным настройкам системы. Для многих европейских стран и России это точка с запятой (;). Если используете запятую (,), можно добавить BOM (Byte Order Mark) для UTF-8, чтобы Excel корректно открыл файл без кракозябр.

python
output = io.StringIO()
output.write('\ufeff')  # BOM для UTF-8
writer = csv.writer(output)
Кодировка: Лучше использовать utf-8-sig (с BOM), но StringIO не поддерживает параметр encoding. Альтернатива — io.BytesIO с ручным кодированием.

Потоковая передача: Если данных много, используйте StreamingResponse с генератором, чтобы не держать весь CSV в памяти.

Как это работает:

Вы отправляете POST-запрос с текстом заметки.

Сервер генерирует уникальный UUID и сохраняет текст в памяти (с таймером удаления).

Вы получаете ссылку вида http://yourapp.com/note/<uuid>.

При переходе по ссылке текст показывается один раз, после чего удаляется с сервера.

python
# app.py
import uuid
import asyncio
from fastapi import FastAPI, HTTPException, Request
from fastapi.responses import HTMLResponse
from pydantic import BaseModel

app = FastAPI(title="One-Time Secret Notes")

# Хранилище в памяти: {uuid: {"text": str, "delete_task": asyncio.Task}}
storage = {}

class NoteCreate(BaseModel):
    text: str
    ttl_minutes: int = 60  # время жизни по умолчанию 60 минут

@app.post("/create", response_model=dict)
async def create_note(note: NoteCreate, request: Request):
    """Создаёт заметку и возвращает ссылку на неё."""
    note_id = str(uuid.uuid4())
    
    # Функция для автоматического удаления заметки через ttl_minutes
    async def delete_after_delay():
        await asyncio.sleep(note.ttl_minutes * 60)
        storage.pop(note_id, None)
    
    # Запускаем таймер на удаление
    task = asyncio.create_task(delete_after_delay())
    storage[note_id] = {"text": note.text, "delete_task": task}
    
    # Формируем полный URL для ответа
    base_url = str(request.base_url).rstrip("/")
    note_url = f"{base_url}/note/{note_id}"
    
    return {"url": note_url, "expires_in_minutes": note.ttl_minutes}

@app.get("/note/{note_id}", response_class=HTMLResponse)
async def read_note(note_id: str):
    """Отображает заметку и немедленно её удаляет."""
    note_data = storage.pop(note_id, None)
    if note_data is None:
        raise HTTPException(status_code=404, detail="Заметка не найдена или уже была прочитана.")
    
    # Отменяем задачу на автоудаление, т.к. мы уже удалили заметку вручную
    note_data["delete_task"].cancel()
    
    # Возвращаем красивую страницу с текстом
    html_content = f"""
    <html>
        <head><title>Секретная заметка</title></head>
        <body style="font-family: sans-serif; max-width: 600px; margin: 50px auto; padding: 20px; border: 1px solid #ccc; border-radius: 8px;">
            <h2>📝 Ваша секретная заметка</h2>
            <div style="background: #f9f9f9; padding: 15px; border-radius: 5px; word-wrap: break-word;">
                {note_data["text"].replace("\n", "<br>")}
            </div>
            <p style="color: #c00; margin-top: 20px;">⚠️ Эта заметка была удалена с сервера и больше не доступна.</p>
            <p><a href="/">Создать новую заметку</a></p>
        </body>
    </html>
    """
    return HTMLResponse(content=html_content)

@app.get("/")
async def root():
    """Простая форма для создания заметок (для удобства)."""
    return HTMLResponse("""
    <html>

                <p>Скопируйте её и передайте адресату. После открытия заметка исчезнет.</p>

import os

TELEGRAM_CHAT_ID = os.getenv("TELEGRAM_CHAT_ID")

async def send_telegram_message(text: str):
    if not TELEGRAM_BOT_TOKEN or not TELEGRAM_CHAT_ID:
        print("Telegram credentials not set, skipping notification.")
        return
    url = f"https://api.telegram.org/bot{TELEGRAM_BOT_TOKEN}/sendMessage"
    async with httpx.AsyncClient() as client:
        await client.post(url, json={"chat_id": TELEGRAM_CHAT_ID, "text": text, "parse_mode": "HTML"})

#### 6. `websocket_manager.py`

```python
from fastapi import WebSocket
from typing import List

class ConnectionManager:
    def __init__(self):
        self.active_connections: List[WebSocket] = []

    async def connect(self, websocket: WebSocket):
        await websocket.accept()
        self.active_connections.append(websocket)

    def disconnect(self, websocket: WebSocket):
        if websocket in self.active_connections:
            self.active_connections.remove(websocket)

    async def broadcast(self, message: dict):
        for connection in self.active_connections:
            try:
                await connection.send_json(message)
            except:
                self.disconnect(connection)

manager = ConnectionManager()
```

#### 7. `monitor.py`

```python
import asyncio
import httpx
from sqlalchemy import select, update
from sqlalchemy.ext.asyncio import AsyncSession
from datetime import datetime, timezone
import logging

from models import Site, CheckLog
from database import AsyncSessionLocal
from telegram_notifier import send_telegram_message
from websocket_manager import manager

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

async def check_site(site: Site, session: AsyncSession):
    start = datetime.now(timezone.utc)
    try:
        async with httpx.AsyncClient(timeout=10.0, follow_redirects=True) as client:
            response = await client.get(str(site.url))
            status = 200 <= response.status_code < 400
            response_time = (datetime.now(timezone.utc) - start).total_seconds() * 1000
            error_msg = None if status else f"HTTP {response.status_code}"
    except Exception as e:
        status = False
        response_time = None
        error_msg = str(e)

    # Сохраняем лог
    log = CheckLog(
        site_id=site.id,
        status=status,
        response_time_ms=response_time,
        error_message=error_msg,
        checked_at=datetime.now(timezone.utc)
    )
    session.add(log)

    # Обновляем информацию о сайте
    site.last_status = status
    site.last_response_time_ms = response_time
    site.last_check_at = datetime.now(timezone.utc)

    # Уведомление при изменении статуса
    old_status = site.last_status
    if old_status != status and old_status is not None:
        emoji = "🟢" if status else "🔴"
        action = "восстановлен" if status else "упал"
        message = f"{emoji} <b>{site.name}</b> {action}!\nURL: {site.url}\nВремя отклика: {response_time:.0f} мс" if status else f"{emoji} <b>{site.name}</b> {action}!\nURL: {site.url}\nОшибка: {error_msg}"
        await send_telegram_message(message)

    await session.commit()

    # Отправляем обновление через WebSocket
    await manager.broadcast({
        "type": "site_update",
        "site_id": site.id,
        "status": status,
        "response_time": response_time,
        "checked_at": site.last_check_at.isoformat()
    })

    return status, response_time

async def run_checks_for_all_sites():
    """Выполняется планировщиком."""
    async with AsyncSessionLocal() as session:
        result = await session.execute(select(Site).where(Site.is_active == True))
        sites = result.scalars().all()
        for site in sites:
            await check_site(site, session)
            await asyncio.sleep(1)  # небольшая пауза между запросами

async def run_checks_loop():
    """Бесконечный цикл для периодических проверок (запускается при старте)."""
    while True:
        await run_checks_for_all_sites()
        await asyncio.sleep(60)  # раз в минуту агрегируем; для индивидуальных интервалов нужна более сложная логика, но для простоты проверяем всех раз в минуту (можно улучшить)
```

#### 8. `app.py` (основной файл FastAPI)

```python
from fastapi import FastAPI, Depends, HTTPException, WebSocket, WebSocketDisconnect
from fastapi.staticfiles import StaticFiles
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select, func, and_
from datetime import datetime, timedelta, timezone
import asyncio
from contextlib import asynccontextmanager

from database import get_db, engine, Base
from models import Site, CheckLog
from schemas import SiteCreate, SiteUpdate, SiteResponse, CheckLogResponse, SiteStats
from monitor import run_checks_loop, check_site
from websocket_manager import manager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Создаём таблицы
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    # Запускаем фоновую задачу проверки сайтов
    task = asyncio.create_task(run_checks_loop())
    yield
    task.cancel()

app = FastAPI(title="Site Monitor Pro", lifespan=lifespan)

# Монтируем статику
app.mount("/static", StaticFiles(directory="static"), name="static")

# ---------- API endpoints ----------
@app.post("/sites", response_model=SiteResponse)
async def create_site(site: SiteCreate, db: AsyncSession = Depends(get_db)):
    db_site = Site(**site.model_dump())
    db.add(db_site)
    await db.commit()
    await db.refresh(db_site)
    return db_site

@app.get("/sites", response_model=list[SiteResponse])
async def list_sites(db: AsyncSession = Depends(get_db)):
    result = await db.execute(select(Site).order_by(Site.created_at.desc()))
    return result.scalars().all()

@app.get("/sites/{site_id}", response_model=SiteResponse)
async def get_site(site_id: int, db: AsyncSession = Depends(get_db)):
    site = await db.get(Site, site_id)
    if not site:
        raise HTTPException(404, "Site not found")
    return site

@app.patch("/sites/{site_id}", response_model=SiteResponse)
async def update_site(site_id: int, site_update: SiteUpdate, db: AsyncSession = Depends(get_db)):
    site = await db.get(Site, site_id)
    if not site:
        raise HTTPException(404, "Site not found")
    update_data = site_update.model_dump(exclude_unset=True)
    for key, value in update_data.items():
        setattr(site, key, value)
    await db.commit()
    await db.refresh(site)
    return site

@app.delete("/sites/{site_id}")
async def delete_site(site_id: int, db: AsyncSession = Depends(get_db)):
    site = await db.get(Site, site_id)
    if not site:
        raise HTTPException(404, "Site not found")
    await db.delete(site)
    await db.commit()
    return {"ok": True}

@app.post("/sites/{site_id}/check")
async def manual_check(site_id: int, db: AsyncSession = Depends(get_db)):
    site = await db.get(Site, site_id)
    if not site:
        raise HTTPException(404, "Site not found")
    status, resp_time = await check_site(site, db)
    return {"status": status, "response_time_ms": resp_time}

@app.get("/sites/{site_id}/logs", response_model=list[CheckLogResponse])
async def get_logs(site_id: int, limit: int = 100, db: AsyncSession = Depends(get_db)):
    result = await db.execute(
        select(CheckLog)
        .where(CheckLog.site_id == site_id)
        .order_by(CheckLog.checked_at.desc())
        .limit(limit)
    )
    return result.scalars().all()

@app.get("/sites/{site_id}/stats", response_model=SiteStats)
async def get_stats(site_id: int, db: AsyncSession = Depends(get_db)):
    site = await db.get(Site, site_id)
    if not site:
        raise HTTPException(404, "Site not found")
    since = datetime.now(timezone.utc) - timedelta(hours=24)
    result = await db.execute(
        select(
            func.count().label("total"),
            func.avg(CheckLog.status.cast(int)).label("uptime_avg"),
            func.avg(CheckLog.response_time_ms).label("resp_avg")
        )
        .where(and_(CheckLog.site_id == site_id, CheckLog.checked_at >= since))
    )
    stats = result.one()
    total = stats.total or 0
    uptime = stats.uptime_avg * 100 if stats.uptime_avg is not None else 0
    avg_resp = stats.resp_avg or 0
    return SiteStats(
        site=SiteResponse.model_validate(site),
        uptime_percentage_24h=uptime,
        avg_response_time_24h=avg_resp,
        checks_count_24h=total
    )

# ---------- WebSocket ----------
@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await manager.connect(websocket)
    try:
        while True:
            await websocket.receive_text()  # keep alive
    except WebSocketDisconnect:
        manager.disconnect(websocket)

# Для удобства отдаём главную страницу
@app.get("/")
async def root():
    from fastapi.responses import FileResponse
    return FileResponse("static/index.html")
```

python
from datetime import datetime, timedelta, timezone
from typing import Optional
from jose import JWTError, jwt
from passlib.context import CryptContext
from pydantic import BaseModel

# Секретный ключ (в production — из переменных окружения)
SECRET_KEY = "your-secret-key-keep-it-secret"
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30
REFRESH_TOKEN_EXPIRE_DAYS = 7

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

class TokenData(BaseModel):
    user_id: Optional[int] = None
    email: Optional[str] = None
    token_type: Optional[str] = None  # "access" или "refresh"

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return pwd_context.verify(plain_password, hashed_password)

def get_password_hash(password: str) -> str:
    return pwd_context.hash(password)

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None) -> str:
    to_encode = data.copy()
    if expires_delta:
        expire = datetime.now(timezone.utc) + expires_delta
    else:
        expire = datetime.now(timezone.utc) + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire, "type": "access"})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def create_refresh_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + timedelta(days=REFRESH_TOKEN_EXPIRE_DAYS)
    to_encode.update({"exp": expire, "type": "refresh"})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def decode_token(token: str) -> Optional[TokenData]:
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        user_id: int = payload.get("sub")
        email: str = payload.get("email")
        token_type: str = payload.get("type")
        if user_id is None:
            return None
        return TokenData(user_id=user_id, email=email, token_type=token_type)
    except JWTError:
        return None

async def get_current_user(
    request: Request,
    db: AsyncSession = Depends(get_db)
) -> Optional[User]:
    """Зависимость, извлекающая пользователя из JWT в Cookie."""
    token = request.cookies.get("access_token")
    if not token:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Not authenticated",
            headers={"WWW-Authenticate": "Bearer"},
        )
    
    token_data = decode_token(token)
    if token_data is None or token_data.token_type != "access":
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid or expired token",
            headers={"WWW-Authenticate": "Bearer"},
        )
    
    result = await db.execute(select(User).where(User.id == token_data.user_id))
    user = result.scalar_one_or_none()
    if user is None or not user.is_active:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="User not found or inactive",
        )
    return user

@app.post("/auth/login")
async def login(
    response: Response,
    form_data: OAuth2PasswordRequestForm = Depends(),
    db: AsyncSession = Depends(get_db)
):
    # Ищем пользователя по email (в форме используется поле username)
    result = await db.execute(select(User).where(User.email == form_data.username))
    user = result.scalar_one_or_none()
    if not user or not verify_password(form_data.password, user.hashed_password):
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Incorrect email or password",
            headers={"WWW-Authenticate": "Bearer"},
        )
    
    # Создаём токены
    access_token = create_access_token(
        data={"sub": user.id, "email": user.email}
    )
    refresh_token = create_refresh_token(
        data={"sub": user.id, "email": user.email}
    )
    
    # Устанавливаем HttpOnly Cookie
    response.set_cookie(
        key="access_token",
        value=access_token,
        httponly=True,
        secure=True,      # только HTTPS (в dev можно False)
        samesite="lax",
        max_age=ACCESS_TOKEN_EXPIRE_MINUTES * 60
    )
    response.set_cookie(
        key="refresh_token",
        value=refresh_token,
        httponly=True,
        secure=True,
        samesite="lax",
        max_age=7 * 24 * 60 * 60
    )
    
    return {
        "success": True,
        "data": {
            "id": user.id,
            "email": user.email,
            "full_name": user.full_name
        }
    }

@app.post("/auth/refresh")
async def refresh_token(
    request: Request,
    response: Response,
    db: AsyncSession = Depends(get_db)
):
    refresh_token_value = request.cookies.get("refresh_token")
    if not refresh_token_value:
        raise HTTPException(status_code=401, detail="Refresh token missing")
    
    token_data = decode_token(refresh_token_value)
    if not token_data or token_data.token_type != "refresh":
        raise HTTPException(status_code=401, detail="Invalid refresh token")
    
    result = await db.execute(select(User).where(User.id == token_data.user_id))
    user = result.scalar_one_or_none()
    if not user or not user.is_active:
        raise HTTPException(status_code=401, detail="User not found")
    
    # Выпускаем новый access token
    new_access_token = create_access_token(
        data={"sub": user.id, "email": user.email}
    )
    response.set_cookie(
        key="access_token",
        value=new_access_token,
        httponly=True,
        secure=True,
        samesite="lax",
        max_age=ACCESS_TOKEN_EXPIRE_MINUTES * 60
    )
    return {"success": True}

@app.post("/auth/logout")
async def logout(response: Response):
    response.delete_cookie("access_token")
    response.delete_cookie("refresh_token")
    return {"success": True, "message": "Logged out"}

🔒 Как это работает с точки зрения безопасности
Токен в HttpOnly Cookie: JavaScript не имеет доступа к токену — XSS-атаки не могут его украсть.

Автоматическая отправка: Браузер сам добавляет Cookie к каждому запросу к вашему API, фронтенду не нужно управлять заголовком Authorization.

Refresh Token: Долгоживущий токен обновления хранится отдельно и используется только на эндпоинте /auth/refresh для получения нового Access Token.

CSRF защита: Благодаря флагу SameSite=lax Cookie не отправляются при кросс-сайтовых POST-запросах, что существенно снижает риск CSRF.

🌐 Интеграция с фронтендом
При использовании Cookie фронтенду не нужно добавлять заголовок Authorization — браузер делает это сам. Пример запроса на защищённый эндпоинт с использованием fetch:

javascript
fetch('/users/me', {
    method: 'GET',
    credentials: 'include' // важно для отправки Cookie при кросс-доменных запросах
})
.then(res => res.json())
.then(data => console.log(data));
💎 Что осталось без JSON?
Логин и рефреш: Возвращают JSON в теле, а токен — в Cookie. Это соответствует требованию «все ответы — JSON».

Ошибки авторизации: Возвращают JSON с success: false и соответствующим HTTP-статусом (401, 403).

Таким образом, мы добавили безопасную и современную авторизацию, не нарушая жёсткое требование о JSON-ответах.

package main

import (
	"regexp"
	"strings"
)

// ValidateEmail проверяет, является ли переданная строка корректным email-адресом.
// Возвращает true, если адрес валиден, и false в противном случае.
func ValidateEmail(email string) bool {
	// 1. Базовая предварительная проверка
	email = strings.TrimSpace(email)
	if len(email) == 0 {
		return false
	}

	// 2. Проверка формата с помощью регулярного выражения
	// Это выражение соответствует спецификации RFC 5322 и покрывает большинство
	// реальных случаев использования, при этом не будучи излишне сложным.
	var emailRegex = regexp.MustCompile(
		`^[a-zA-Z0-9.!#$%&'*+/=?^_` + "`" + `{|}~-]+@` + // локальная часть
			`[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?` + // домен (второй уровень)
			`(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$`, // домен (верхний уровень и поддомены)
	)
	if !emailRegex.MatchString(email) {
		return false
	}

	// 3. Дополнительная проверка на длину (RFC 5321)
	if len(email) > 254 {
		return false
	}

	// 4. Проверка, что локальная часть не слишком длинная (RFC 5321)
	parts := strings.Split(email, "@")
	if len(parts) != 2 || len(parts[0]) > 64 {
		return false
	}

	// 5. Проверка, что доменная часть не содержит последовательных точек
	if strings.Contains(parts[1], "..") {
		return false
	}

	// 6. Проверка, что доменная часть не начинается и не заканчивается дефисом
	if strings.HasPrefix(parts[1], "-") || strings.HasSuffix(parts[1], "-") {
		return false
	}

	return true
}
3. Стоит ли вкладывать все сбережения в биткоин прямо сейчас?

Как ИИ-ассистент, я не уполномочен давать индивидуальные инвестиционные рекомендации, но могу выделить ключевые факторы, которые вам следует учесть при принятии самостоятельного решения:

Чрезвычайно высокая волатильность. Биткоин исторически демонстрирует колебания цены на десятки процентов за короткий срок. Такая волатильность делает его непригодным для хранения всех сбережений, особенно если эти средства могут понадобиться вам в обозримом будущем.

Принцип диверсификации. Это базовое правило инвестирования: «не кладите все яйца в одну корзину». Концентрация всего капитала в одном, даже самом перспективном, активе неоправданно увеличивает риск безвозвратной потери средств.

Отсутствие гарантий и защиты. Биткоин — это децентрализованный цифровой актив. В отличие от банковских вкладов, он не застрахован государством. Потеря доступа к вашему приватному ключу, взлом биржи или мошенничество могут привести к полной и безвозвратной утере ваших сбережений.

Спекулятивный характер. Значительная часть цены биткоина определяется не его внутренней полезностью, а рыночными ожиданиями и настроениями. Это делает его цену непредсказуемой и подверженной резким обвалам.

Регуляторные риски. Отношение правительств разных стран к криптовалютам постоянно меняется. Ужесточение регулирования может негативно сказаться на цене и доступности биткоина.

Прежде чем принимать какие-либо решения, особенно касающиеся всего вашего капитала, настоятельно рекомендую вам проконсультироваться с квалифицированным финансовым советником. ⚛️ От бита к кубиту: принцип суперпозиции
Классический бит в любой момент времени находится в одном из двух состояний: 0 или 1. Кубит (квантовый бит) — это его обобщение, которое благодаря законам квантовой механики может находиться в суперпозиции — одновременно и в состоянии 0, и в состоянии 1.

Математически это состояние описывается вектором в двумерном комплексном пространстве (сфере Блоха) и выглядит так:
|ψ⟩ = α|0⟩ + β|1⟩

|ψ⟩ (пси) — это обозначение квантового состояния кубита.

|0⟩ и |1⟩ — базисные состояния, аналоги классических 0 и 1.

α (альфа) и β (бета) — комплексные числа, которые называются амплитудами вероятности.

Ключевой момент: пока кубит находится в суперпозиции, мы не можем сказать, в каком конкретно состоянии он находится. Но при измерении кубита его суперпозиция необратимо разрушается, и он «схлопывается» в одно из базисных состояний — |0⟩ с вероятностью |α|² или |1⟩ с вероятностью |β|².

🧬 Физическая реализация кубитов
Создать кубит на практике можно разными способами. У каждого из них есть свои преимущества и недостатки:

Сверхпроводящие кубиты: Самый популярный и коммерчески успешный тип. В их основе лежат искусственные атомы — сверхпроводящие цепи с джозефсоновскими контактами, работающие при температурах, близких к абсолютному нулю (-273°C). Их ключевое преимущество — возможность массового производства на чипах с использованием литографии, как и классических процессоров.

Ионные кубиты: Используют заряженные атомы (ионы), удерживаемые в вакууме электромагнитными полями. Их квантовое состояние кодируется в энергетических уровнях электронов. Они отличаются очень высокой точностью операций и долгим временем жизни квантового состояния.

Фотонные кубиты: Информацию переносят фотоны — элементарные частицы света. Их главное преимущество в том, что они могут передаваться на большие расстояния и слабо взаимодействуют с окружением, что снижает количество ошибок.

Топологические кубиты: Это перспективная, но пока экспериментальная технология. Информация в них кодируется не в состоянии отдельной частицы, а в глобальных, «топологических» свойствах всей системы. Ожидается, что такие кубиты будут обладать естественной защитой от помех.

🎛️ Квантовые вентили: инструменты для вычислений
Для выполнения операций над кубитами используются квантовые вентили. Это аналоги классических логических вентилей (И, ИЛИ, НЕ), но они описывают преобразования вектора состояния кубита. Математически каждый вентиль — это унитарная матрица, на которую умножается вектор состояния кубита или системы кубитов.

Однокубитные вентили:
Они меняют состояние одного кубита, «вращая» его вектор на сфере Блоха.

Вентиль Паули-X (или NOT): Аналог классического «НЕ». Он меняет состояние на противоположное: |0⟩ превращает в |1⟩, и наоборот.

Вентиль Адамара (H): Один из самых важных вентилей. Он создает равновероятную суперпозицию. Если применить его к кубиту в состоянии |0⟩, он перейдет в состояние (|0⟩ + |1⟩)/√2, где вероятности получить 0 или 1 при измерении равны 50%.

Многокубитные вентили:
Они создают связь между кубитами и являются основой для квантового параллелизма. Наиболее важный — двухкубитный вентиль CNOT (управляемое НЕ). Он действует на два кубита: управляющий и управляемый. Если управляющий кубит находится в состоянии |1⟩, то вентиль CNOT применяет NOT к управляемому кубиту; если в состоянии |0⟩ — оставляет его без изменений.

Именно такие многокубитные вентили позволяют создавать квантовую запутанность — уникальное явление, при котором состояния двух или более кубитов становятся взаимосвязанными, даже если их разделяют огромные расстояния.

🗺️ Универсальный набор вентилей и квантовые схемы
Для выполнения произвольного квантового алгоритма не нужны все возможные вентили. Достаточно универсального набора вентилей, который способен аппроксимировать любое унитарное преобразование с любой заданной точностью.

Стандартный универсальный набор включает:

Однокубитные вентили (например, вентиль Адамара H и вентиль сдвига фазы T).

Хотя бы один запутывающий двухкубитный вентиль, такой как CNOT.

Последовательность таких вентилей образует квантовую схему — своего рода «программу» для квантового компьютера. Именно из таких схем строятся все квантовые алгоритмы, включая знаменитые алгоритмы Шора (для факторизации чисел) и Гровера (для поиска в неструктурированной базе данных).

🛡️ Проблема ошибок и квантовая коррекция
Главный враг квантовых вычислений — декогеренция, то есть потеря кубитом своего хрупкого квантового состояния из-за взаимодействия с окружающей средой. Это приводит к ошибкам в вычислениях. Простое копирование информации, как в классических компьютерах, здесь невозможно из-за фундаментальной теоремы о запрете клонирования.

Решение — квантовая коррекция ошибок. Один из самых многообещающих методов — поверхностный код (surface code). Его идея заключается в следующем:

Один логический кубит (который и используется в алгоритме) кодируется состоянием множества физических кубитов (например, 17 или 49), расположенных на двумерной решетке.

Часть этих кубитов используется для хранения данных, а другая — для обнаружения ошибок без прямого измерения самих данных, что разрушило бы их квантовое состояние.

Чем больше физических кубитов в логическом, тем надежнее он защищен от ошибок. Недавние эксперименты, в том числе от Google Quantum AI, впервые показали, что масштабирование такого кода выгодно на практике — увеличение числа физических кубитов уменьшает вероятность ошибки.

Эта технология — ключевой шаг на пути к созданию отказоустойчивого квантового компьютера, способного решать задачи, недоступные любым современным суперкомпьютерам.

💎 Заключение
Таким образом, работа квантового компьютера — это тонкое искусство управления квантовыми состояниями. Оно начинается с кубита, способного к суперпозиции, использует квантовые вентили для запутывания и манипуляции этими состояниями, и, наконец, полагается на сложные методы коррекции ошибок для получения надежных результатов. Вся эта экосистема и лежит в основе вычислительной мощности нового поколения.

💰 Финансовая сторона: Во что это обойдется?
Важно понимать, что стоимость SOC 2 — это не только и не столько плата аудитору, но и целый комплекс расходов на подготовку.

Аудит и подготовка: Для небольшого стартапа полная стоимость подготовки и прохождения аудита SOC 2 Type I обычно составляет от $30,000 до $50,000. Более подробная разбивка затрат может выглядеть так:

Аудит Type I: ~$8,000 – $12,000.

Подготовка (gap-анализ): ~$10,000 – $25,000.

Инструменты автоматизации (Vanta, Drata): $6,000 – $25,000 в год.

Пентест: $3,000 – $20,000.

Юридическая поддержка: $5,000 – $10,000.

Внутренние трудозатраты: ~100-500+ часов работы вашей команды.

Альтернатива для старта — SOC 2 Type II: Хотя Type I дешевле и быстрее, большинство клиентов ожидают SOC 2 Type II. Type II — это не просто снимок состояния на конкретный момент, а подтверждение того, что ваши процессы безопасности эффективно работают на протяжении определенного периода (обычно 3-12 месяцев). Инвестиции здесь выше ($20,000 – $50,000+), но такая сертификация дает гораздо более сильный сигнал рынку и закрывает вопрос для большинства клиентов.

Регулярные расходы: Сертификат SOC 2 Type II требует ежегодного обновления, бюджет на поддержание которого составляет примерно $10,000 – $40,000 в год.

⏳ Временные рамки: Когда вы получите заветный отчет?
SOC 2 Type I: Самый быстрый путь. При должной подготовке можно уложиться в 4-12 недель.

SOC 2 Type II: Это марафон. Процесс включает в себя период наблюдения, поэтому весь цикл от старта до получения отчета занимает от 4 до 12 месяцев. Однако некоторые провайдеры с помощью автоматизации обещают сроки в 4.5-6 месяцев.

🤝 Переговоры с клиентом: Как разделить риски и затраты?
Это ключевой момент. Не стоит брать на себя все риски в одиночку.

Начните диалог: Объясните клиенту, что вы небольшой стартап, но готовы инвестировать в безопасность и соответствие их требованиям. Подчеркните, что SOC 2 — это длительный процесс.

Предложите компромиссные варианты:

Промежуточный аудит (Gap Assessment): Предложите провести его сейчас, а полную сертификацию SOC 2 Type II — позже. Это дешевле и быстрее, а результаты можно показать клиенту как доказательство вашей работы над безопасностью.

Письмо о намерениях (Letter of Intent): Подпишите документ, в котором вы обязуетесь пройти сертификацию в обмен на заключение контракта.

Частичная компенсация: Попросите клиента частично профинансировать или компенсировать стоимость аудита, особенно если контракт долгосрочный.

Пилотный проект: Предложите начать с небольшого пилотного проекта, который не требует полной сертификации, чтобы доказать свою надежность.

🛠️ Инструменты: Ваш главный союзник в борьбе с рутиной
Для команды из трех разработчиков ручное управление compliance просто нереалистично. К счастью, есть инструменты, которые автоматизируют до 80% этой работы.

Vanta: Популярный выбор среди стартапов благодаря простоте использования и быстрой настройке.

Drata: Предлагает глубокую автоматизацию и сильные возможности для совместной работы с аудитором.

Delve (AI-first): Фокусируется на скорости и ручной поддержке, используя AI для автоматизации сбора доказательств.

Comp AI (Open Source): Бесплатная альтернатива для автоматизации compliance с SOC 2.

Использование таких платформ может сократить время подготовки на 40-60%.

💡 Альтернативы: Что, если не SOC 2?
Возможно, клиенту нужны не конкретно SOC 2, а просто гарантии безопасности. В таком случае можно предложить альтернативы:

ISO 27001: Международный стандарт, более комплексный и признанный в Европе. Если ваш клиент — международная компания, это может быть даже более весомым аргументом.

Сторонний пентест и аудит безопасности: Проведите его силами авторитетной компании. Отчет от известного аудитора может быть достаточным доказательством вашей безопасности.

💎 Резюме: Практический план действий
Уточните требования: Спросите клиента, нужен ли им именно SOC 2 Type II. Если да, переходите к следующему шагу.

Выберите инструмент автоматизации: Это сэкономит десятки (если не сотни) часов работы вашей команды.

Заключите контракт с аудиторской фирмой: Выбирайте фирму, которая имеет опыт работы со стартапами и понимает вашу специфику.

Запустите процесс, параллельно ведя переговоры: Используйте факт начала подготовки к SOC 2 как козырь в переговорах с клиентом.

Управлять монорепозиторием — это как управлять огромным, сложным механизмом. Google пошла на это, вложив огромные ресурсы в создание собственной, уникальной экосистемы для работы с кодом:

Своя система контроля версий: Piper/CitC. Это не просто Git, а целый комплекс для работы с петабайтами данных, который, среди прочего, позволяет разработчикам редактировать код "в облаке", не скачивая весь репозиторий.

Огромные инвестиции: Тысячи инженеров годами создавали и поддерживают эту инфраструктуру.

Единый источник правды: Любой инженер может легко найти и изучить код любого другого сервиса.

Атомарные изменения: Можно одной правкой обновить общую библиотеку и сразу все 1000 сервисов, которые ее используют.

Упрощенное переиспользование кода: Нет необходимости публиковать внутренние библиотеки в пакетный менеджер.

🫀 Шаг 1. Heartbeat — как лидер остаётся «живым»
Механизм предельно прост:

Лидер шлёт пустые AppendEntries. Сразу после победы на выборах новый лидер начинает рассылать всем фолловерам пустые RPC-вызовы AppendEntries. Это и есть сердцебиение (heartbeat) — способ лидера заявить: «Я жив, я ваш лидер, не начинайте новых выборов».

Фолловеры сбрасывают таймер. Как только фолловер получает такой heartbeat, он немедленно сбрасывает свой внутренний election timer в ноль. Пока лидер шлёт сигналы чаще, чем истекает этот таймер, в кластере тишина и спокойствие.

«Тишина» запускает выборы. Если фолловер в течение всего election timeout (о них — ниже) не получает ни одного heartbeat, он делает логичный вывод: «Лидер, видимо, упал». После этого он переходит в состояние Candidate и начинает новые выборы.

🧠 Шаг 2. Split-brain — почему Raft не «сходит с ума»
В Raft вы в принципе не можете получить ситуацию «двух лидеров, принимающих записи». Безопасность данных гарантирована математически — принципом кворума.

Что такое split-brain в распределённых системах?

Представьте, что кластер из 5 серверов временно распался на две изолированные группы (например, из-за проблем с сетью): в первой — 2 узла, во второй — 3. Если каждая группа сможет выбрать «своего» лидера, то у вас появятся два центра принятия решений. Они начнут независимо принимать и фиксировать изменения от клиентов. Это и есть split-brain (раскол мозга). Когда связь восстановится, объединить две расходящиеся истории изменений будет практически невозможно, и данные окажутся испорчены.

Как Raft решает проблему?

Raft не даёт изолированной группе из 2 узлов выбрать лидера в принципе, потому что действует железное правило: для победы на выборах кандидат должен получить голоса от строгого большинства (majority) всех узлов кластера.

В кластере из 5 узлов «большинство» — это как минимум 3 голоса. Группа из 2 узлов физически не может набрать кворум, даже если все они проголосуют «за». В результате в меньшей части кластера выборы будут бесконечно проваливаться, а записи от клиентов — блокироваться.

Да, доступность системы в этой части упадёт (клиенты не смогут писать), но данные останутся консистентными. Жертвуем доступностью ради согласованности — классический выбор CP-системы в терминах CAP-теоремы.
